## Overview

The Notification Configs API manages automatic notification configurations for user documents and aircraft services. Cronjobs evaluate these configurations daily and send notifications when documents are approaching expiration or services are due.

## Authentication

All endpoints require authentication via JWT cookie (`cookieAuth`).

## Base URL

```
/api/notification-configs
```

---

## Create Notification Config

Create a notification configuration for a user document or aircraft service.

<CodeGroup>
```bash User Document Notification
curl -X POST https://api.sigsaho.com/api/notification-configs \
  -H "Content-Type: application/json" \
  -b "cookies.txt" \
  -d '{
    "notification_time": "FIFTEEN_DAYS_BEFORE",
    "target_type": "USER_DOC",
    "user_doc_id": 5,
    "is_active": true
  }'
```

```bash Aircraft Service Notification
curl -X POST https://api.sigsaho.com/api/notification-configs \
  -H "Content-Type: application/json" \
  -b "cookies.txt" \
  -d '{
    "notification_time": "SEVEN_DAYS_BEFORE",
    "target_type": "AIRCRAFT_SERVICE",
    "aircraft_service_id": 12,
    "is_active": true
  }'
```
</CodeGroup>

### Request Body

<ParamField body="notification_time" type="string" required>
  When to send the notification relative to the expiration/scheduled date.
  
  **Options:**
  - `ON_EXPIRATION_DAY` - Send on the day of expiration/service
  - `ONE_DAY_BEFORE` - Send 1 day before
  - `SEVEN_DAYS_BEFORE` - Send 7 days before
  - `FIFTEEN_DAYS_BEFORE` - Send 15 days before
</ParamField>

<ParamField body="target_type" type="string" required>
  Type of entity this notification applies to.
  
  **Options:**
  - `USER_DOC` - User document notification
  - `AIRCRAFT_SERVICE` - Aircraft service notification
</ParamField>

<ParamField body="is_active" type="boolean" default="true">
  Whether the configuration is active and should be evaluated by cronjobs
</ParamField>

<ParamField body="user_doc_id" type="integer">
  ID of the user document. **Required when `target_type` is `USER_DOC`**
</ParamField>

<ParamField body="aircraft_service_id" type="integer">
  ID of the aircraft service. **Required when `target_type` is `AIRCRAFT_SERVICE`**
</ParamField>

### Response

<ResponseField name="message" type="string">
  Success message
</ResponseField>

<ResponseField name="notificationConfig" type="object">
  The created notification config object
  
  <Expandable title="NotificationConfig properties">
    <ResponseField name="id" type="number">
      Configuration ID
    </ResponseField>
    <ResponseField name="notification_time" type="string">
      Notification timing
    </ResponseField>
    <ResponseField name="target_type" type="string">
      Target entity type
    </ResponseField>
    <ResponseField name="is_active" type="boolean">
      Active status
    </ResponseField>
    <ResponseField name="sent_at" type="string" nullable>
      Timestamp when notification was sent (null if not sent)
    </ResponseField>
    <ResponseField name="user_doc" type="object" nullable>
      Related user document (present if target_type is USER_DOC)
    </ResponseField>
    <ResponseField name="aircraft_service" type="object" nullable>
      Related aircraft service (present if target_type is AIRCRAFT_SERVICE)
    </ResponseField>
    <ResponseField name="created_at" type="string">
      Creation timestamp
    </ResponseField>
    <ResponseField name="updated_at" type="string">
      Last update timestamp
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseExample>

```json 201 Success
  "message": "Configuración de notificación creada exitosamente",
  "notificationConfig": {
    "id": 1,
    "notification_time": "SEVEN_DAYS_BEFORE",
```

```json 400 Validation Error
{
  "error": "La combinación (target_type + FK + notification_time) ya existe"

```

```json 401 Unauthorized
{
  "error": "No autorizado"
```

### Validation Rules

1. **Target Type Validation:**
   - If `target_type` is `USER_DOC`, `user_doc_id` must be provided
   - If `target_type` is `AIRCRAFT_SERVICE`, `aircraft_service_id` must be provided

2. **Uniqueness:**
   - The combination of `(target_type, FK, notification_time)` must be unique
   - Cannot create duplicate configurations for the same entity and timing

3. **Foreign Key Validation:**
   - The referenced `user_doc_id` or `aircraft_service_id` must exist

## Update Notification Config

Update an existing notification configuration's timing or active status.

<CodeGroup>
```bash PUT /notification-configs/:id
curl -X PUT https://api.sigsaho.com/api/notification-configs/1 \
  -H "Content-Type: application/json" \
  -b "cookies.txt" \
  -d '{
    "notification_time": "ONE_DAY_BEFORE",
    "is_active": false
  }'
```
</CodeGroup>

### Path Parameters

<ParamField path="id" type="integer" required>
  Notification config ID
</ParamField>

### Request Body

All fields are optional. Provide only the fields you want to update.

<ParamField body="notification_time" type="string">
  New notification timing: `ON_EXPIRATION_DAY`, `ONE_DAY_BEFORE`, `SEVEN_DAYS_BEFORE`, or `FIFTEEN_DAYS_BEFORE`
</ParamField>

<ParamField body="is_active" type="boolean">
  Set to `false` to disable the notification (cronjob will ignore it)
</ParamField>

### Response

<ResponseField name="message" type="string">
  Success message
</ResponseField>

<ResponseField name="notificationConfig" type="object">
  Updated notification config object
</ResponseField>

<ResponseExample>
```json 200 Success
{
  "message": "Configuración de notificación actualizada exitosamente",
  "notificationConfig": {
    "id": 1,
    "notification_time": "ONE_DAY_BEFORE",
    "target_type": "USER_DOC",
    "is_active": false,
    "sent_at": null,
    "created_at": "2024-01-15T10:00:00.000Z",
    "updated_at": "2024-03-15T14:20:00.000Z"
  }
}
```
</ResponseExample>

### Important Notes

<Warning>
  **Cannot Change Target:** You cannot modify `target_type`, `user_doc_id`, or `aircraft_service_id`. To change the target entity, delete the configuration and create a new one.
</Warning>

**Required Permission:** Write access to Mantenimiento module

## Target Types

### USER_DOC
Notifications for user document expirations (license, visa, passport, psycho-physical aptitude, RATARI).

**Evaluated Against:** Document `expiration_date` field

**Example Use Cases:**
- Pilot license renewals
- Visa expiration warnings
- Medical certificate expirations

### AIRCRAFT_SERVICE
Notifications for scheduled aircraft maintenance services.

**Evaluated Against:** Service `scheduled_date` field

**Example Use Cases:**
- 100-hour inspections
- Annual maintenance
- Component replacement schedules

## Cronjob Behavior

The notification cronjob runs daily and:

1. **Filters active configs:** Only evaluates configs where `is_active = true` and `sent_at = null`
2. **Calculates trigger date:** Determines the notification trigger date based on `notification_time`
3. **Checks current date:** Compares today's date with the calculated trigger date
4. **Sends notification:** If trigger date matches, sends the notification
5. **Updates sent_at:** Sets `sent_at` to the current timestamp after successful send

### Trigger Date Calculation

```
Trigger Date = Expiration/Scheduled Date - Notification Offset

Examples:
- ON_EXPIRATION_DAY: March 25 - 0 days = March 25
- ONE_DAY_BEFORE: March 25 - 1 day = March 24
- SEVEN_DAYS_BEFORE: March 25 - 7 days = March 18
- FIFTEEN_DAYS_BEFORE: March 25 - 15 days = March 10
```

## Best Practices

<AccordionGroup>
  <Accordion title="Multiple Reminder Strategy" icon="layer-group">
    Create multiple configurations with different timings for critical documents:
    
    - 15 days before (initial warning)
    - 7 days before (follow-up)
    - 1 day before (urgent reminder)
    - On expiration day (critical alert)
  </Accordion>
  
  <Accordion title="Reset After Updates" icon="rotate">
    Always reset `sent_at` when updating the expiration date or scheduled date of the related document/service to ensure notifications are sent again.
  </Accordion>
  
  <Accordion title="Disable Instead of Delete" icon="toggle-off">
    Use `is_active: false` instead of deleting configs when temporarily disabling notifications. This preserves the configuration history.
  </Accordion>
  
  <Accordion title="Monitor Sent Notifications" icon="chart-line">
    Regularly query configs with `sent=true` to verify notifications are being sent as expected.
  </Accordion>
</AccordionGroup>

## Example Workflows

### Workflow 1: Configure License Expiration Notification

```bash
# 1. Create notification for 15 days before expiration
curl -X POST https://api.sigsaho.com/api/notification-configs \
  -H "Content-Type: application/json" \
  -b "cookies.txt" \
  -d '{
    "notification_time": "FIFTEEN_DAYS_BEFORE",
    "target_type": "USER_DOC",
    "user_doc_id": 42,
    "is_active": true
  }'

# 2. Add follow-up notification for 1 day before
curl -X POST https://api.sigsaho.com/api/notification-configs \
  -H "Content-Type: application/json" \
  -b "cookies.txt" \
  -d '{
    "notification_time": "ONE_DAY_BEFORE",
    "target_type": "USER_DOC",
    "user_doc_id": 42,
    "is_active": true
  }'
```

### Workflow 2: Update Service Date and Reset Notification

```bash
# 1. Update service scheduled date (via Aircraft Services API)
curl -X PUT https://api.sigsaho.com/api/aircraft-services/12 \
  -H "Content-Type: application/json" \
  -b "cookies.txt" \
  -d '{"scheduled_date": "2024-05-15T00:00:00.000Z"}'

# 2. Reset notification to trigger new send
curl -X PATCH https://api.sigsaho.com/api/notification-configs/5/reset \
  -b "cookies.txt"
```
